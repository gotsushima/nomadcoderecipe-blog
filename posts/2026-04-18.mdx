## 今日の問い

Supabase の Row Level Security と Next.js の Server Components をどう組み合わせるか——この設計判断を Claude と ChatGPT の両方に投げた。同じプロンプト、同じコンテキスト。返ってきた回答には、予想以上の差があった。

## ChatGPT の回答

即座に、クリーンなコードを返してきた。

export async function getPost(id: string) {
  const supabase = createServerComponentClient({ cookies })
  const { data } = await supabase
    .from('posts')
    .select('*')

## Claude の回答

Claude は先にこう問い返してきた。

> 「この `posts` テーブルには誰がアクセスできる想定ですか？認証済みユーザーのみ？それとも公開データもある？RLS のポリシーを先に決めないと、Server Component 側の実装方針も変わります」

そして2パターンの実装を提示した。

**パターン A：公開記事 + 下書きが混在する場合**

```sql
-- Supabase RLS ポリシー
create policy "公開記事は誰でも読める"
  on posts for select
  using (status = 'published');

create policy "下書きは本人のみ"
  on posts for select
  using (auth.uid() = author_id and status = 'draft');
```

```typescript
// anon key で接続 → RLS が自動でフィルタリング
// Server Component から呼ぶ場合
export async function getPublishedPost(slug: string) {
  const supabase = createClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY! // anon key で十分
  )
  const { data } = await supabase
    .from('posts')
    .select('*')
    .eq('slug', slug)
    .eq('status', 'published')
    .single()
  return data
}
```

**パターン B：完全非公開データ (今回のブログの conversations テーブル)**

```typescript
// service_role key を使う → RLS をバイパス
// サーバーサイド限定。クライアントに絶対漏らさない
export async function getConversations(date: string) {
  const supabase = createClient(
    process.env.SUPABASE_URL!,
    process.env.SUPABASE_SERVICE_ROLE_KEY! // サーバーのみ
  )
  return supabase
    .from('conversations')
    .select('*')
    .eq('date', date)
    .eq('processed', false)
}
```

## 差の正体

ChatGPT は「何を作るか」に答えた。Claude は「なぜそう作るか」から始めた。

| | ChatGPT | Claude |
|---|---|---|
| 回答スタイル | 実装を即提示 | 前提を確認してから提示 |
| RLS の扱い | 省略 | 設計の核として扱う |
| セキュリティ | 実装者が考慮する前提 | 設計に組み込む |
| 複数パターン | 1つの正解 | ユースケース別に分岐 |
